# CLAUDE.md - Kernel design system

## Quick orientation

- **Components:** `src/components/` (generic atoms) + `src/components/chat/` (./chat tier) + `src/components/clinical/` (./clinical tier)
- **Tokens:** `tokens/tokens.css` (DTCG, generated - do not hand-edit)
- **Standard:** `STANDARD.md` / `standard.json` - the single prop-vocabulary source of truth
- **Catalog:** `catalog.json` (generated by `npm run catalog`) - machine-readable index of all 81 components

## Key commands

```bash
npm run gate           # regenerate catalog.json + run the catalog gate (must be green before commit)
npm run test           # run the CLI + MCP test suite (node --test, 36 tests)
npm run storybook      # start Storybook at http://localhost:6006
npm run gen-stories    # regenerate baseline story files from catalog.json (skips existing)
npm run build-storybook # build the static gallery
```

## Authoring components - the workflow

**ALWAYS** run these before writing markup:

1. `kernel component <Name>` - check if a component already exists
2. `kernel search <concept>` - find related components

**NEVER** re-author `data-*` state as props; never use raw `<button>` / `<input>` instead of Kernel atoms.

## Storybook MCP on-ramp

Storybook (`npm run storybook`) exposes an MCP server at `http://localhost:6006/mcp`.
It is registered in `.mcp.json` and activates when Storybook is running.

**ALWAYS query the Storybook MCP before authoring or editing a story** - never hallucinate
component props. Available tools (once `npm run storybook` is running):

- `list-all-documentation` - browse all components and their stories
- `get-documentation` - get props / argTypes / story code for a component
- `get-documentation-for-story` - get the exact rendered story code
- `get-storybook-story-instructions` - authoring instructions for a specific story
- `get-changed-stories` - see what changed since the last test run
- `preview-stories` - render a story in the browser
- `run-story-tests` - run vitest/a11y tests against stories

**Workflow for story authoring:**
1. Start Storybook: `npm run storybook`
2. Query the MCP: `list-all-documentation` to find existing stories
3. `get-documentation <ComponentName>` to read real props before writing JSX
4. Write/edit the story file in `src/stories/ComponentName.stories.jsx`
5. `run-story-tests` to confirm green

## Story conventions (Astryx-modeled - astryx.atmeta.com)

The Storybook is organized like Astryx: a nested sidebar, and per component a
**Playground** sandbox (all props as controls) + a **Gallery** hero, with the Docs
page carrying usage / Do-Don't / anatomy. Author stories to match.

- **Title tree = `tools/lib/taxonomy.mjs`** - the single source of truth for the sidebar.
  Scheme: `{Tier}/{Category}/{Family?}/{Name}` (no `Kernel/` prefix - the whole book is
  Kernel). Tiers: **Core** (`.`) / **Chat** (`./chat`) / **Clinical** (`./clinical`).
  Variant/part families nest under a parent (e.g. `Core/Overlays/Menu/MenuItem`). Never
  hand-write a `title:` - it is derived; run the retitle tool if the taxonomy changes.
- **Per component, prefer the Playground pattern** over many single-purpose stories:
  - **`Playground`** - every prop wired as a control (the Astryx "Properties" sandbox).
    This is how a reviewer explores combinations; do NOT split `Selected`/`Muted`/`WithIcon`
    into separate stories - make them controls. (See `Chip` / `Button` as the exemplars.)
  - **`Gallery`** - one render of the meaningful variant matrix (the Astryx "Overview" hero).
  - Add named **Examples** only for genuinely-distinct scenarios (real data, composition).
- **Baseline generation:** `npm run gen-stories` projects a Playground + Gallery + rich
  autodocs from `catalog.json` (`--force` refreshes generated files; `@hand-authored`
  files are never overwritten). `node tools/retitle-stories.mjs` re-titles every file
  (hand-authored included) to the taxonomy and drops deprecated-alias stories.
- **Deprecated aliases get no story** (`Btn`/`PAv`/`Tip`) - they are noted on the
  canonical component (`Button`/`PersonAvatar`/`Tooltip`) instead. See `taxonomy.mjs`.
- **Decorator:** every story is auto-wrapped in `<div class="krnl-companion-layer" data-brand
  data-theme data-density data-breakpoint>` (`.storybook/preview.jsx`).
- **Toolbar globals:** Brand (corilus/semble/myneva), Theme (light/dark), Density,
  Breakpoint - QA across the full matrix.

**Before authoring a story:** query the Storybook MCP (`get-documentation <Name>`) AND/OR
`kernel component <Name>` for the real props - never hallucinate. After: `run-story-tests`.

## Storybook structure

```
.storybook/
  main.js          # SB config, addons (addon-mcp, addon-a11y, addon-docs, addon-vitest)
  preview.jsx      # globals, decorator, CSS imports
  react-global.js  # window.React/ReactDOM shim (MUST be first in preview.jsx)
tools/
  lib/taxonomy.mjs   # THE sidebar tree: tier + category + variant-nesting (single source)
  gen-stories.mjs    # baseline Playground/Gallery/autodocs generator (from catalog.json)
  retitle-stories.mjs# re-title every story to the taxonomy; drop alias stories
src/stories/
  ComponentName.stories.jsx   # 78 stories (81 catalog entries minus 3 deprecated aliases)
```
